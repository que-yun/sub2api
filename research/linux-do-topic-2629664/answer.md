# Sub2API 缓存诊断（2026-07-21）

## 结论

当前本机 Sub2API 的调度缓存没有出现 outbox 积压或版本快照膨胀，但存在明显的永久账号快照残留：Redis 中有 18,442 份 `sched:acc:*` 和同量 `sched:meta:*`，数据库仅有 4,237 个未软删除账号。14,197 份缓存对应已软删除账号，另有 8 份数据库已不存在。该类键没有 TTL，并占据 Redis 字符串数据的大部分。

运行中的 Redis 当前约 282 MiB、74,943 个键，未设置 `maxmemory`，策略为 `noeviction`。继续积累时会优先面临宿主内存压力，而不是可控淘汰。

`user_refresh_tokens:1` 另有 63,867 个成员、约 7 MiB，且当前没有 TTL。现行写入代码会在每次写入时为整个集合续期，因此持续活跃用户的旧 token 索引不会自然按成员过期；但本次观测到的 `TTL=-1` 与该正常写入路径不完全一致，还需追查历史版本或绕过路径。

## 已核实

- 调度快照版本在切换后只给旧版本 60 秒宽限期，不是内存增长主因。
- 调度 outbox 只有 3 行，Redis watermark 等于数据库最大事件 ID；认证缓存失效 outbox 无待处理记录。因此目前没有观察到配置变更传播积压。
- 当前代码的账号删除路径会尝试删除对应调度缓存；现存残留更可能是历史数据、旧版本路径或曾经失败且未补偿的删除。
- 调度账号快照是完整 `Account` 序列化，残留可能继续保存软删除账号的凭据。

## 建议动作

1. 先做一次受控清理：按 `sched:acc:<id>` 扫描并以数据库当前存在且未软删除的账号作为白名单，成对删除 `sched:acc:<id>` 与 `sched:meta:<id>`。不要 `FLUSHDB`，也不要删除活跃快照或并发控制键。
2. 清理前后记录键数和 `used_memory`；清理后验证调度 outbox、水位、网关请求和账号删除流程。
3. 追查 `user_refresh_tokens:1` 的无 TTL 来源；随后为 `user_refresh_tokens:*` / `token_family:*` 增加成员级过期或定期清扫，避免集合因持续续期无限增长。
4. 为 Redis 设容量预算与告警；选定淘汰策略前需确认可接受 refresh token 被淘汰后要求重新登录的行为。不要只设 `maxmemory` 且保留 `noeviction`。
